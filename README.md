\## Теоретична частина

\### REST API

REST (Representational State Transfer) - архітектурний стиль взаємодії між клієнтом та сервером через протокол HTTP. Ключові принципи REST:

\- \*\*Клієнт-серверна архітектура\*\* - чіткий розподіл відповідальності між клієнтом (UI) та сервером (логіка, дані).

\- \*\*Відсутність стану (Stateless)\*\* - кожен запит є самодостатнім і містить усю необхідну інформацію.

\- \*\*Уніфікований інтерфейс\*\* - стандартизоване використання HTTP-методів: `GET`, `POST`, `PUT`, `DELETE`.

\- \*\*Адресація ресурсів через URI\*\* - кожен ресурс має унікальну адресу (наприклад, `/api/energy-storage/1`).

HTTP-методи та їх семантика:

| Метод | Призначення | Ідемпотентність |

|----------|------------------------------------|:---------------:|

| `GET` | Отримання даних | ✅ |

| `POST` | Створення або відправка команди | ❌ |

| `PUT` | Повне або часткове оновлення | ✅ |

| `DELETE` | Видалення ресурсу | ✅ |

Стандартні коди відповідей HTTP:

| Код | Значення |

|-------|----------------------------------|

| `200` | OK - запит виконано успішно |

| `201` | Created - ресурс створено |

| `400` | Bad Request - некоректний запит |

| `404` | Not Found - ресурс не знайдено |

| `500` | Internal Server Error |

\### Node.js + Express

\*\*Node.js\*\* - серверне JavaScript-середовище на основі движка V8. Особливості:

\- Асинхронна, подієво-орієнтована модель виконання.

\- Єдина мова (JavaScript) для клієнта та сервера.

\- Вбудований пакетний менеджер npm.

\*\*Express.js\*\* - мінімалістичний веб-фреймворк для Node.js. Надає зручний механізм опису маршрутів та формування JSON-відповідей:

```javascript

const express = require('express');

const app = express();

app.use(express.json()); // Парсинг тіла запиту у форматі JSON



app.get('/api/resource', (req, res) => {

&#x20;   res.json({ data: \[] });

});



app.listen(3000);

```

\### Формат JSON

\*\*JSON (JavaScript Object Notation)\*\* - текстовий формат обміну даними, заснований на підмножині синтаксису JavaScript. Основні типи значень: рядок (`string`), число (`number`), булеве значення (`boolean`), масив (`array`), об'єкт (`object`), `null`.

Приклад JSON-об'єкта BESS-системи:

```json

{

&#x20; "id": 1,

&#x20; "name": "Станція-1",

&#x20; "capacity": 5000,

&#x20; "stateOfCharge": 92,

&#x20; "mode": "discharging"

}

```

Переваги JSON: зручний формат, нативна підтримка у JavaScript, широка підтримка в усіх мовах програмування.

\### Опис об'єкта автоматизації

Об'єктом автоматизації є \*\*батарейна система накопичення енергії (BESS - Battery Energy Storage System)\*\* - промисловий комплекс, призначений для акумулювання електричної енергії з метою балансування навантаження в енергомережі, резервного живлення або інтеграції відновлюваних джерел енергії.

BESS виконує три основні операційні режими:

\- \*\*Charging (заряд)\*\* - накопичення енергії з мережі або від генератора.

\- \*\*Discharging (розряд)\*\* - віддача накопиченої енергії в мережу або споживачам.

\- \*\*Idle (очікування)\*\* - система підключена, але не виконує активного обміну енергією.

Параметри об'єкта:

| Параметр | Тип | Опис |

|-------------------|----------|-------------------------------------------|

| `id` | `number` | Унікальний ідентифікатор BESS |

| `name` | `string` | Назва системи накопичення |

| `capacity` | `number` | Загальна ємність системи (кВт·год) |

| `power` | `number` | Максимальна потужність (кВт) |

| `stateOfCharge` | `number` | Поточний рівень заряду (%) |

| `chargingPower` | `number` | Фактична потужність заряду (кВт) |

| `dischargingPower`| `number` | Фактична потужність розряду (кВт) |

| `cycleCount` | `number` | Загальна кількість циклів заряд/розряд |

| `temperature` | `number` | Температура батарейного блоку (°C) |

| `mode` | `string` | Режим: `charging` / `discharging` / `idle`|

\### Перелік вимог

\*\*Функціональні вимоги:\*\*

1\. Отримання переліку всіх BESS-систем.

2\. Отримання детальної інформації про конкретну систему за її `id`.

3\. Отримання поточного рівня заряду (SOC) конкретної системи.

4\. Відправка команди управління режимом роботи системи.

5\. Оновлення параметрів конкретної системи.

6\. Видалення системи з реєстру.

\*\*Нефункціональні вимоги:\*\*

\- Відповіді у форматі JSON.

\- Коректна обробка помилок (404, 400) з описовими повідомленнями.

\- Статичне обслуговування клієнтського інтерфейсу сервером.

\- Інтерактивний браузерний інтерфейс для тестування всіх endpoints.

\---

\### Обґрунтування вибраних методів реалізації

\*\*Node.js + Express\*\* обрано через:

\- Мінімальний поріг входу при наявності знань JavaScript.

\- Вбудовані засоби для роботи з HTTP та JSON (`express.json()`).

\- Зручний декларативний опис маршрутів.

\*\*In-memory зберігання даних\*\* (масив `bessSystems`) обрано для спрощення прототипу без необхідності налаштування бази даних.

\*\*Статичне обслуговування\*\* через `express.static(\_\_dirname)` дозволяє розгортати клієнтський інтерфейс (`index.html`, `style.css`, `script.js`) без окремого веб-сервера.

\*\*REST-архітектура\*\* обрана як де-факто стандарт для промислових API завдяки простоті, масштабованості та широкій підтримці у клієнтських інструментах.

\## Опис реалізації

\### Завдання 1: Проєктування API

\#### Аналіз предметної галузі

\*\*Основні сутності:\*\*

Єдина центральна сутність системи - `BessSystem` (система накопичення енергії). Система є самодостатнім об'єктом, що містить усі оперативні та конфігураційні параметри.

\*\*Параметри для відстеження:\*\*

\- \*\*Ємнісні:\*\* `capacity` (кВт·год) - незмінна характеристика.

\- \*\*Оперативні:\*\* `stateOfCharge`, `chargingPower`, `dischargingPower`, `mode` - змінюються в реальному часі.

\- \*\*Діагностичні:\*\* `temperature`, `cycleCount` - показники стану батареї.

\- \*\*Конфігураційні:\*\* `name`, `power` - адміністративні дані.

\*\*Взаємозв'язки між даними:\*\*

```

stateOfCharge  ←→  mode (charging/discharging/idle)

&#x20;                      ↕

&#x20;              chargingPower / dischargingPower

temperature    ←→  cycleCount (деградація батареї)

```

Значення `mode` визначає, яке з полів потужності активне: при `charging` - `chargingPower > 0`, при `discharging` - `dischargingPower > 0`, при `idle` - обидва дорівнюють нулю.

\#### Проєктування структури API

\*\*Схема endpoints:\*\*

```

GET    /api/energy-storage           → Список усіх систем

GET    /api/energy-storage/:id       → Система за ID

GET    /api/energy-storage/:id/soc   → Рівень заряду системи

POST   /api/energy-storage/:id/command → Команда управління

PUT    /api/energy-storage/:id       → Оновлення параметрів

DELETE /api/energy-storage/:id       → Видалення системи

```

\*\*Структура JSON-об'єкта (повна модель):\*\*

```json

{

&#x20; "id": 1,

&#x20; "name": "Станція-1",

&#x20; "capacity": 5000,

&#x20; "power": 2500,

&#x20; "stateOfCharge": 92,

&#x20; "chargingPower": 0,

&#x20; "dischargingPower": 500,

&#x20; "cycleCount": 15,

&#x20; "temperature": 24,

&#x20; "mode": "discharging"

}

```

\*\*Коди відповідей:\*\*

| Ситуація | Код |

|----------------------------------|------|

| Успішне отримання даних | 200 |

| Система не знайдена | 404 |

| Відсутнє тіло команди (POST/PUT) | 400 |

| Системна помилка сервера | 500 |

\#### Реалізація серверної частини (server.js)

Сервер реалізовано на Express.js. Дані зберігаються в оперативній пам'яті у вигляді масиву об'єктів `bessSystems`. Для пошуку системи за ID використовується допоміжна функція `findBess`:

```javascript
const findBess = (id) => bessSystems.find((b) => b.id === parseInt(id));
```

Особливість реалізації `PUT` - злиття наявного об'єкта з даними з тіла запиту через оператор spread, із примусовим збереженням оригінального `id`:

```javascript

bessSystems\[index] = { ...bessSystems\[index], ...req.body, id: parseInt(req.params.id) };

```

Endpoint `/soc` реалізує просту бізнес-логіку - формування статусу на основі порогового значення заряду:

```javascript
status: system.stateOfCharge < 20 ? "Low Battery" : "Normal";
```

\#### Реалізація клієнтського інтерфейсу (script.js, index.html, style.css)

Клієнтська частина реалізована як односторінковий застосунок (SPA) з трьома функціональними блоками:

1\. \*\*Блок тестування API\*\* - набір кнопок для виклику кожного endpoint. ID системи задається через числове поле введення.

2\. \*\*Блок поточних даних\*\* - відображає актуальний стан усіх систем у форматі JSON (оновлюється автоматично після кожного модифікуючого запиту).

3\. \*\*Блок відповіді сервера\*\* - показує HTTP-статус та тіло відповіді останнього виклику.

Центральна функція `apiCall` є універсальним обгортником для `fetch`:

```javascript

async function apiCall(method, url, bodyData = null) {

&#x20;   const options = { method };

&#x20;   if (bodyData) {

&#x20;       options.headers = { 'Content-Type': 'application/json' };

&#x20;       options.body = JSON.stringify(bodyData);

&#x20;   }

&#x20;   const response = await fetch(url, options);

&#x20;   const data = await response.json();

&#x20;   // оновлення UI...

&#x20;   if (method !== 'GET') updateDbView();

}

```

Кольорове кодування статусів: зелений фон - успіх (`2xx`), червоний - помилка (`4xx`/`5xx`). Стиль інтерфейсу виконано в темній кольоровій схемі з акцентним кольором `#00f2fe`.
